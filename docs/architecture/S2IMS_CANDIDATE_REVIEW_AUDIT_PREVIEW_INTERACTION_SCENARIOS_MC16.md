# S²IMS Candidate Review Audit Preview Interaction Scenarios MC16

## Purpose

This document defines interaction scenarios for future runtime QA of the MC15 diagnostic preview UI.

The scenarios are planning-only. They do not implement UI changes, audit writes, persistence, backend/API calls, assignment, approval, AP-10B approval collection, AP-10C, or AP-11.

## Scenario List

Future runtime QA should cover:
- initial render
- empty diagnostic preview state
- local review actions
- diagnostic preview generation
- clear/reset behavior
- repeated action behavior
- switching candidate/action behavior
- readonly behavior
- keyboard accessibility
- screen-reader understandability
- negative behavior safety checks

## Initial State Scenarios

| Scenario | Expected Result |
|----------|-----------------|
| Load shell with no generated preview | Empty state copy is visible |
| Load shell with candidates | No candidate is auto-selected |
| Load shell with candidate actions available | No action is triggered automatically |
| Load shell in readonly mode | Controls are disabled and communicate disabled state |

## Action State Scenarios

| Action | Expected Diagnostic Result | Safety Boundary |
|--------|----------------------------|-----------------|
| Shortlist | Preview describes local shortlist signal | Not saved, not approval, not assignment |
| Skip | Preview describes local skip signal | Not scholarship rejection |
| Needs more context | Preview describes local context request signal | Not task creation, not notification |
| Reject for assignment | Preview describes assignment-unsuitable signal | Not scholarship rejection |
| Select for review | Preview describes manual review selection signal | Not assignment, not approval |

Every generated preview must keep visible:
- `Diagnostic preview`
- `Local UI signal only`
- `Not saved`
- `Not submitted`
- `Not official evidence`
- `Not an approval`
- `Not an assignment`

## Clear / Reset Scenarios

| Scenario | Expected Result | Must Not Happen |
|----------|-----------------|-----------------|
| Clear diagnostic preview | Empty state returns | No server deletion |
| Clear after several actions | Only local preview is removed | No audit deletion |
| Clear in readonly mode | No enabled clear action should appear unless preview state exists and local-only behavior is clear | No persistence mutation |

## Repeated-Action Scenarios

| Scenario | Expected Result | Must Not Happen |
|----------|-----------------|-----------------|
| Click same action repeatedly | Latest preview remains clear and local-only | No duplicate persisted record |
| Switch from shortlist to skip | Preview reflects latest local signal | No hidden approval or assignment |
| Switch candidates | Preview reflects latest candidate/action context | No default selected candidate |

## Readonly Scenarios

Readonly QA should verify:
- action controls are disabled
- disabled controls include accessible disabled semantics
- no local preview is generated by disabled controls
- copy does not imply approval or assignment
- no keyboard path can activate disabled controls

## Accessibility Scenarios

Future runtime QA should verify:
- tab order follows visible layout
- focus states are visible
- warning and preview regions are readable by screen readers
- `aria-live="polite"` or equivalent preview-status announcement is present
- badge meanings are visible text
- false flags are visible text
- color is not the only carrier of state

## Negative Behavior Scenarios

Future runtime QA should confirm no:
- audit writes
- backend/API calls
- browser storage writes
- export/download
- notification
- route transition
- Staff callback
- official evidence creation
- assignment
- approval
- scholarship decision
- AP-10B approval collection
- AP-10B gate change

## Expected Evidence to Capture During Future Runtime QA

Capture:
- screenshots of empty state
- screenshots of preview state
- screenshot or transcript of visible false flags
- keyboard navigation notes
- screen-reader notes or accessibility inspection notes
- route smoke results
- dev log grep result
- static safety grep result for forbidden behavior
- confirmation that diff scope excludes runtime changes unless future runtime QA is explicitly approved
