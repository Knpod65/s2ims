01

Design vision

What the product must feel like — and what it must never feel like

S²IMS is an institutional matching platform that handles sensitive student data, scholarship money, and provider relationships. The interface is the trust layer. It must feel premium and futuristic, but never theatrical. It must feel intelligent, but never opaque. It must feel safe, but never bureaucratic.

The "Tactile Future" aesthetic

Hybrid of Glassmorphism (transparency, lightness, modernity) and a soft Neumorphic lift (light upper-left + cool lower-right shadows on critical surfaces) on a soft, warm-cool off-white canvas. The result is a "gallery" feel — quiet, premium, considered. Role color is wayfinding, not decoration.

It should feel

  • Premium — generous whitespace, restrained color, considered type
  • Futuristic — glass, blur, soft depth, pulsing freshness indicators
  • Human-centered — rounded shapes, plain language, friendly empty states
  • University-grade — institutional weight, AA+ contrast, audit-aware
  • Privacy-aware — masking is visible, identity is earned not assumed
  • Data/AI platform — match scores, fairness, freshness, confidence as first-class UI

It must not feel

  • Like a generic admin dashboard — no SaaS-template purple gradients on white cards
  • Childish — no emojis as UI, no playful illustrations in core flows
  • Cyberpunk / over-dark — no neon-on-black; this is a daylight system
  • Bureaucratic — no compliance-form aesthetic, no hostile dense forms
  • Toy-like — no oversized rounded corners on functional elements, no bouncy springs

Three principles, in order

  1. Trust before delight. If a delight choice (animation, gradient, micro-interaction) competes with audit clarity or identity protection, drop the delight.
  2. Role identity is wayfinding. A user must know which role they are operating as within 1 second of any screen, without reading a label. Color, gradient direction, and badge anchor this.
  3. Governance is part of the product, not a layer on top. Audit warnings, reason fields, masked identities, and disclosure approvals are first-class components with the same polish as KPIs and tables.

Role-based visual system

Five themes — each a complete visual identity, not just a color

Every screen in S²IMS belongs to exactly one role at a time. Role identity is set at the AppShell level via the data-role attribute and propagates through CSS variables. Role color is wayfinding — used for sidebar active state, primary buttons, KPI accents, role badge, progress rings, and selection states. It is never used for full backgrounds, large surface fills, or hero panels.

06

Governance UX

Non-negotiable. The trust layer of the product.
⚖
Inviolable rules. These are visual and behavioral commitments Claude Code must enforce at the component level — not as comments, not as toggles. Violations are bugs.
  1. Provider never sees student identity (name, photo, email, ID number) before disclosure approval.
  2. Provider candidate pool uses Candidate #C-XXXX tokens only.
  3. Staff sees Student #S-XXXX by default; reveal requires explicit reason + audit entry.
  4. Executive views are aggregate-only. No row-level student data, ever.
  5. Admin audit views show student tokens by default; un-masking requires reason + 2-step.
  6. Sensitive actions (export, override, role change, manual match) require a typed reason.
  7. Matching override requires a reason — and writes both before/after match scores to audit.
  8. Role changes require a reason and a confirmation step naming both old and new role.
  9. Export actions surface the audit-warning card before the export button is enabled.
  10. Sync retry writes an audit log entry with operator, target system, and timestamp.

ตัวอย่างภาษาไทย: ระบบจับคู่ทุนการศึกษาอัจฉริยะ — บรรทัดความสูง 1.7 ทำให้สระและวรรณยุกต์ไม่ทับกัน

Button labels

  • Min button width: 88px desktop, 64px mobile — Thai labels frequently 1.6× English. Buttons may grow but never wrap to two lines on desktop.
  • Mobile wrapping: if Thai label exceeds button width, allow 2-line wrap with line-height: 1.3 and reduce button radius to keep tap target geometry. Never truncate with ellipsis on primary actions.
  • EN ↔ TH lengths: design every screen with the longer of EN × 1.6 or TH. Never lock widths to EN copy.

Tables & cards with Thai text

  • Table cells use line-height: 1.7 when any cell contains Thai. Apply at the row level, not the column.
  • Numeric and ID columns stay LTR; Thai labels in headers respect normal reading flow (Thai is LTR but visually denser).
  • Cards: title font-size unchanged across languages; card height grows rather than truncating.
  • Truncation is never on critical info (student name, scholarship name, deadline). Truncate only on description/preview.

TH/EN toggle behavior

  • Segmented control in topbar; both labels always visible (no current-language hiding).
  • Toggle is instant — no full reload. Only language strings rebind. Persisted server-side.
  • Numbers: Thai locale uses Thai digits only when explicitly set; default is Arabic numerals for clarity in financial contexts.
  • Dates: DD MMM YYYY format — month name localized, year is BE (พ.ศ.) when locale = TH, AD when EN. Show both on first hover for ambiguous dates.
i
Mixed-script lines: when Thai and English mix in one paragraph, set font-family to a stack starting with the EN family — browsers fall back to TH for Thai codepoints automatically. Don't force per-span swaps.

10

Handoff to Claude Code

The exact, actionable rules. Implement in this order. Skip nothing.

Implementation rules

  1. Tokens first. Generate tokens.css from §02 before any component. Never hard-code colors, radii, or shadows in components.
  2. Role theming via attribute selector. One data-role on the shell rebinds --role-primary, --role-primary-2, --role-gradient, --role-tint. Components reference these vars only.
  3. Build the shell before any screen. AppShell is the integration test for tokens, theming, language, freshness, and permissions.
  4. Build governance components before dashboards. MaskedIdentityCard, CandidateTokenLabel, AggregateOnlyBadge, AuditWarningCard, ReasonRequiredModal, DisclosureApprovalNotice, FairnessAlert, DataFreshnessIndicator, PermissionDenied.
  5. Identity gates are component-level, not page-level. A provider screen cannot import an unmasked student card — make this a TypeScript boundary.
  6. Reason fields are validated. Min 20 chars, trimmed, persisted with action audit entry. Submit disabled until valid.
  7. One PermissionDenied component. Used for every RBAC failure. Reveals only the resource type.
  8. Mobile-first transformations. Implement the desktop ↔ mobile pairs from §07 as discriminated UI variants, not separate pages.
  9. TH locale is wired from day one, even if strings are placeholders. Line-height swaps and IBM Plex Sans Thai Looped (display) + Noto Sans Thai (body) are loaded by the shell.
  10. Accessibility: AA+ contrast, 44px tap target, focus-visible rings, prefers-reduced-motion respected for the freshness pulse and modal entrances.

Notes Claude Code must follow

  • Use Tailwind only as a utility convenience layer — all design tokens come from CSS variables. No tailwind.config color extends; reference vars via theme('colors.role.primary') alias if needed.
  • Every component file lives next to a .stories.tsx (or equivalent) covering: empty / loading / error / role-themed (5 variants) / TH locale.
  • Governance components are not generic — they are a separate folder with a CODEOWNER. Changes here require a paired audit-spec entry.
  • Mock data must use real Token formats (S-YYYY-NNNN, C-YYYY-NNNN, SCH-XXX-NN, PRV-NNN). Never use names like "John Doe" outside disclosed contexts.
  • Loading: skeleton-first, no spinners on dashboards. Skeletons mirror the layout they replace.
  • Animations respect prefers-reduced-motion — disable freshness pulse and reduce modal slide to a fade.
  • Focus-visible rings use --role-primary; never browser default.
  • Errors never reveal sensitive data. "Couldn't load" + retry — never the underlying record name.

Visual QA checklist

A reviewer should walk through this for every screen before merge

Identity & theme

  • Role badge visible in topbar; matches data-role on shell.
  • Sidebar active item uses --role-tint bg + --role-primary text + 3px gradient bar.
  • Primary buttons use --role-gradient; no hard-coded role hexes anywhere.
  • No role color appears as a full background fill or large hero panel.
  • Page title set to Page · Role · S²IMS.

Governance

  • Provider screens: zero student names, photos, or emails. Tokens only (C-YYYY-NNNN).
  • Staff screens: Student tokens (S-YYYY-NNNN) shown by default.
  • Executive screens: Aggregate-Only Dashboard badge present at the top.
  • Admin audit: tokens by default; reveal requires reason and writes audit entry.
  • All sensitive actions (override, export, role change, disclosure) gate behind ReasonRequiredModal with min-20-char validation.
  • Audit warning card precedes every destructive or recorded action.
  • Sync retry writes audit log (verify with mock audit trail).

Responsive

  • Tested at 375 / 768 / 1280 / 1920 widths.
  • Tables collapse to mobile data cards below 768.
  • Drawers become full-screen routes on mobile.
  • Filter panel becomes bottom sheet on mobile, with apply count in CTA.
  • Touch targets ≥ 44px on every tappable element on mobile.
  • Sticky CTA sits ≥ 48px above bottom nav and respects safe-area inset.

Internationalization

  • TH/EN toggle is in topbar, both labels visible, no reload on switch.
  • Tested with Thai labels at 1.6× English length — no critical truncation.
  • Thai body text uses 1.7 line-height; row heights grow as needed.
  • Dates show BE year for TH, AD for EN.
  • IDs (S-…, C-…) remain ASCII mono in both locales.

States

  • Empty / loading / error states use the shared component, not bespoke text.
  • Skeletons mirror layout — no spinner-only loads on dashboards.
  • Permission denied uses the shared component and never reveals beyond resource type.
  • Freshness indicator shows on every integrated KPI / list (SIS, SSO, Provider sync).

Accessibility

  • AA+ contrast on all text against glass surfaces.
  • Focus-visible ring uses --role-primary, 2px offset.
  • prefers-reduced-motion: pulse disabled, modal entrance simplified.
  • All interactive elements reachable by keyboard in DOM order.
  • Screen-reader labels on icon-only buttons (notifications, profile, sort).
